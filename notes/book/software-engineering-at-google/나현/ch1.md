# Ch1. 소프트웨어 엔지니어링이란?

## 프로그래밍 vs 소프트웨어 엔지니어링

- 프로그래밍에 **시간**이라는 중요한 차원이 더해져 더 입체적으로 바뀐다
- 정육면체는 정사각형이 아니고, 거리는 속도가 아니듯 소프트웨어 엔지니어링은 프로그래밍이 아니다
- 둘의 차이는 소프트웨어의 **지속 가능성(sustainability)** 에 있다
- `"이 코드의 예상 수명은?"` 이라는 질문 하나가 그 차이를 드러낸다

> **"나 혼자, 지금만 돌아가면 끝" → 프로그래밍**
> **"팀이, 시간이 지나도 고칠 수 있게" → 소프트웨어 엔지니어링**

```
소셜 로그인 다중 플랫폼 통합 시, 플랫폼별로 if-else 분기 처리하면 당장은 돌아간다.
그런데 플랫폼이 추가되거나 API 스펙이 바뀌는 순간 그 코드는 "시간의 무게"를 못 버티고 무너진다.
Factory Pattern으로 설계하는 것이 바로 시간 차원을 고려한 것 — 코드의 수명을 늘린 선택이다.
```

## 시간과 변경

- 수명이 길어질수록 변경이라는 요소가 점점 중요해진다
- 업그레이드에 성공하든 중도에 포기하든, 이런 고통을 한 번 겪고 나면 겁먹고 '다시는 시도하지 않겠다'고 다짐하기도 한다
- 이런 결론에 이른 조직은 기존 코드를 버리고 새로 작성하거나 업그레이드를 완전히 포기해버리곤 한다
- 하지만 고통을 회피하는 방안보다 고통을 덜어줄 방법을 찾는 것이 때로는 더 합리적인 선택이다
- 지속 가능하게 하려면 요구되는 변경들의 영향을 계획하고 관리해야 한다
- 수명이 길어질수록 **'동작한다'** 와 **'유지보수 가능하다'** 의 차이를 더 분명하게 인지해야 한다

### 기대 수명 × 업그레이드 비용

어느 선택이 더 옳을지는 전적으로 **업그레이드 비용 / 업그레이드가 가져다줄 이익 / 프로젝트의 기대 수명**에 달려있다.

Spring Boot 2.x는 2024년 11월 공식 지원 종료(EOL). 즉 보안 패치가 나오지 않는다.
팀에서 결정을 내려야 한다: **Spring Boot 3.x로 올릴 것인가, 말 것인가?**

Spring Boot 2 → 3은 단순 버전업이 아니다. 실제로 부딪히는 것들:
- `javax.*` → `jakarta.*` 패키지 전체 교체 (JPA, Servlet 포함)
- Spring Security 설정 방식이 완전히 바뀜
- 일부 third-party 라이브러리가 3.x 미지원일 수 있음

고통스럽다. 그래서 팀이 "그냥 2.x 쓰자, 지금 잘 돌아가잖아"라고 결론 내리는 게 책이 말하는 상황이다.

| 프로젝트 기대 수명 | 업그레이드 결정 | 이유 |
|---|---|---|
| 6개월 안에 종료 예정 | 안 해도 됨 | 보안 이슈 터지기 전에 서비스 종료됨 |
| 2~3년 이상 운영 예정 | 해야 함 | 보안 취약점 누적, 나중엔 더 큰 비용 발생 |
| 수명 불명확 | 해두는 게 맞음 | 불확실성 자체가 "길게 간다"는 신호 |

```
Spring Boot 2.x → "지금 동작한다" ✅
Spring Boot 2.x → "3년 뒤에도 유지보수 가능하다" ❌

핵심 질문: "이 결정은 6개월 뒤의 나 혹은 팀원이 고마워할 결정인가?"
```

## 하이럼의 법칙 (Hyrum's Law)

> API 사용자가 충분히 많다면 API 명세에 적힌 내용은 중요하지 않다.
> 시스템에서 눈에 보이는 모든 행위를 누군가는 이용하게 될 것이기 때문이다.

- 최선의 의도, 최고의 엔지니어, 꼼꼼한 코드 리뷰가 뒷받침되더라도 공표한 계약이나 모범 사례를 완벽하게 구현해냈다고 단정할 수 없다
- API 소유자는 인터페이스를 명확하게 설명해놓으면 어느 정도의 유연성과 자유를 얻을 수 있다
- 하지만 현실에서는 사용자가 명세에 없는 기능을 찾아 활용하기도 하며, 그 기능이 널리 쓰이면 추후 API를 변경하기 어렵게 된다
- 따라서 변경이 얼마나 유용할지를 분석할 때는 이러한 충돌을 조사·식별·해결하는 데 따르는 비용도 고려해야 한다

```java
// 명세: "userId로 사용자 조회, 없으면 null 반환"
public User findUser(Long userId) {
	return userRepository.findById(userId).orElse(null);
}
```

명세엔 null 반환이라 적혀있다. 근데 이 API를 1년간 쓰다 보면 어느 순간 이런 코드가 생긴다.

```javascript
// "null이면 비로그인 유저로 처리하는 로직"
if (response.data === null) showGuestUI();
```

그런데 API 개발자가 null 대신 예외를 던지도록 수정하면?
명세에 없던 null 동작을 누군가 이미 믿고 있었기 때문에 터진다.

```
사람은 원래 명세를 끝까지 안 읽는다. 동작이 편리하면 그냥 쓴다.
시간이 지나 새로 합류한 사람은 "원래 이렇게 동작하는 거 아니야?" 하고 그냥 믿어버린다.
결국 "되는 것"은 암묵적 계약이 된다.
```

| 구분 | 기발한 코드 | 클린한 코드 |
|---|---|---|
| 특징 | 명세 밖의 동작을 이용 | 명세 안에서만 움직임 |
| 단기 | 빠르게 목적 달성 | 약간 돌아가는 느낌 |
| 장기 | 의존하던 동작이 바뀌면 폭탄 | 상대 API가 바뀌어도 안전 |
| 적합한 상황 | 수명 짧은 스크립트, 빠른 검증 | 장기 운영 서비스 |

```
"기발한"이 칭찬으로 느껴지면 프로그래밍, "질책"으로 느껴지면 소프트웨어 엔지니어링.
내부 공통 라이브러리나 플랫폼 API에 의존하는 코드를 짤 때,
공식 문서에 없는 동작을 슬쩍 이용하는 순간 그게 언제 터질지 모르는 시한폭탄이 된다.
그걸 PR에서 잡아주는 문화가 있는 곳이 좋은 회사다.
```

## 지속 가능성에 투자하지 않는 장기 프로젝트가 위험한 이유

### 1. 보안 구멍
의존하는 라이브러리 EOL 이후 CVE(보안 취약점)가 발견돼도 패치가 나오지 않는다. 구멍 뚫린 채로 운영하게 된다.

### 2. 기반 기술 변화
MySQL 5.7도 2023년 EOL이다. 8.0으로 올리면 쿼리 옵티마이저가 바뀌어서 열심히 튜닝해놓은 인덱스 전략이 오히려 더 느려지는 케이스가 생길 수 있다. 최적화가 버전에 종속되어 있기 때문이다.

### 3. 효율 개선의 불확실성
```
옛날 최적화 = "DB 조회를 줄여라" → 캐싱 열심히
지금 최적화 = "DB 조회를 줄여라" + "Redis 있으면 네트워크 레이턴시가 변수"
내일 최적화 = ??? (하드웨어/인프라 변화에 따라 다름)
```

그 시절 정답이 지금도 정답이란 보장이 없다. N+1 잡아서 쿼리 줄였는데, 서비스 규모가 커지면 오히려 Join보다 분리 조회가 나은 상황이 오기도 한다. 적절한 변경이 뒷받침되어야 새로운 선택지의 도입 효과가 극대화된다.

## 규모 확장과 효율성

- 코드베이스의 수명이 다할 때까지 직면하는 모든 변경을 안전하게 처리할 수 있다면 그 코드베이스는 지속 가능하다
- 변경 비용이 시간 흐름보다 가파르게 상승하는 시스템은 확장 가능하지 않다
- 비용이 많이 드는 변경은 지연되기 쉽고, 시간이 흐르다 보면 결국 예상치 못한 변경이 불가피한 시점이 온다

```
변경 비용
    │         ❌ 확장 불가능한 시스템
    │        /  (변경할수록 더 아파짐)
    │       /
    │      /  ✅ 지속 가능한 시스템
    │    ──    (변경 비용이 완만하게 유지됨)
    │
    └──────────────────────────────── 시간
```

변경 비용이 가파르게 오르는 시스템은 어느 순간 "건드리면 안 되는 코드"가 생기기 시작한다.
"이 부분은 왜 이렇게 짜여있는지 아무도 모르니까 건드리지 말자"

### "인력을 두 배로 늘리면 되지 않을까?"

인력 충원은 선형적으로 확장되지만, 협업 비용은 비선형적으로 증가한다.

- 커뮤니케이션 비용이 기하급수적으로 늘어남
- 코드베이스를 파악해야 하는 온보딩 비용 발생
- 서로 다른 부분을 건드리다 충돌 발생

코드베이스 자체도 확장 가능해야 한다:

- **빌드 시간**: 모듈이 20개 넘어가면 전체 빌드가 5~10분씩 걸리기 시작한다. 코드 한 줄 고치고 결과 보는 데 10분 기다려야 하면 하루 작업 사이클이 확 줄어든다
- **테스트 시간**: 테스트 코드가 쌓일수록 전체 테스트 실행에 30분~1시간이 걸린다. CI/CD 파이프라인이 병목이 돼버리면 배포 자체가 느려진다
- **버전 관리**: 커밋 이력이 수십만 개 쌓이면 `git log`, `git blame` 하나 돌리는 것도 느려지기 시작한다

그래서 모듈 분리, 빌드 캐싱, 테스트 병렬화 등으로 조직 차원에서 챙겨야 한다.

### 확장하기 어려운 정책들

조직이 커질수록 다음 질문을 스스로 던져봐야 한다:

- 조직이 10배로 커지면 이 작업도 10배로 많아지는가?
- 코드베이스가 커질수록 작업량도 늘어나는가?
- 자동화하거나 최적화할 수단이 있는가?
- 마지막 질문의 답이 '아니오'라면 확장성에 문제가 있는 것

### 비욘세 규칙 — 확장 가능한 정책의 예

> "인프라를 변경하여 서비스가 중단되는 등의 문제가 발생하더라도,
> 같은 문제가 CI 시스템의 자동 테스트에서 발견되지 않는다면 인프라팀의 책임이 아니다"
> = **네가 좋아했다면 CI 테스트를 준비해뒀어야지**

```
// 규칙이 없는 세상
인프라팀: "MySQL 8.0으로 올려도 될까요?"
서비스팀 A: "저희 괜찮을 것 같아요"
서비스팀 B: "음... 잘 모르겠어요"
서비스팀 C: "저희 코드 한번 봐줄 수 있어요?"
인프라팀: (서비스 10개 전부 수동 검토 시작)

// 비욘세 규칙이 있는 세상
인프라팀: MySQL 8.0으로 올립니다. CI에서 터지는 건 각 팀 책임.
서비스팀 A: (CI 통과) ✅
서비스팀 B: (CI 실패) ❌ → 자기 코드 고침
서비스팀 C: (CI 실패) ❌ → 자기 코드 고침
인프라팀: (서비스 내부는 신경 안 써도 됨)
```

서비스가 10개 → 100개가 돼도 인프라팀 인원은 그대로여도 된다. 이게 선형보다 나은 이유다.

```
테스트 코드를 "버그 잡으려고 쓰는 것"으로만 생각하기 쉬운데,
비욘세 규칙 관점에서 보면 테스트 코드 = 내 서비스가 외부 변경으로부터 스스로를 지키는 보호막이다.
```

## 트레이드오프와 비용

- 비용은 단순히 돈이 아니라 다음을 모두 포괄한다
    - 금전적 비용, 리소스 비용 (CPU 등), 인적 비용, 거래 비용, 기회 비용, 사회적 비용
- "좋은 결정"이 아니라 **"비용을 알고 내린 결정"** 이 낫다
- 기술 부채를 쌓더라도, 그게 얼마짜리 부채인지 알고 쌓는 팀이 모르고 쌓는 팀보다 훨씬 건강하다

```
인덱스 튜닝 시 "지금 당장 쿼리를 고쳐도 되지만, 실제로는 테이블 설계 자체를 바꿔야 근본 해결"이라는 걸
알면서도 배포 일정, 데이터 마이그레이션, 연관 영향도 때문에 연기하고 인덱스로 임시 처리하는 상황이 온다.

이 임시 결정을 기록해두고 나중에 반드시 재검토하는 것이 "지연 비용을 정확히 계산한다"는 말의 의미다.
구글은 이것을 코드 리뷰, 기술 부채 트래킹 등 시스템으로 관리한다.
```

### 왼쪽으로 이동하기 (Shift Left)

버그를 오른쪽(프로덕션)에서 발견할수록 비용이 기하급수적으로 올라간다.

```
코드 작성 → 코드 리뷰 → 빌드 → 테스트 → 스테이징 → 프로덕션
  (왼쪽)                                              (오른쪽)
```

| 발견 시점 | 비용 |
|---|---|
| 코드 작성 중 (IDE 경고) | 5분 |
| 코드 리뷰 | 30분 |
| 테스트 실패 | 2시간 |
| 스테이징 | 반나절 |
| 프로덕션 | 며칠 + 장애 보고서 |

정적 분석 도구, 린트, 단위 테스트가 전부 이 맥락이다.

## 정리

- **시간(Time)**: "지금 동작한다" ≠ "내일도 유지보수 가능하다". 기대 수명이 모든 기술 결정의 기준이 되어야 한다
- **규모(Scale)**: 사람이 반복하는 모든 작업은 자동화 대상이다. 코드베이스 자체도 확장 가능해야 한다
- **트레이드오프(Trade-offs)**: 완벽한 결정은 없다. 비용을 알고 결정하고, 지연된 결정의 비용도 기록해두어야 한다

```
소프트웨어 엔지니어링은 프로그래밍에 시간과 규모가 더해진 것이다.
"이 코드를 3년 뒤 모르는 사람이 고쳐야 한다면?" — 이 질문을 습관적으로 던지는 것이 시작이다.
```
